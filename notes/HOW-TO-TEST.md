```
## 작업 순서

### 1단계: 코드 파악
먼저 shop.py, prices.json, preferences_*.md, trend_*.md를 모두 읽고
앱의 실제 동작을 이해해. 추측하지 말고 코드 기반으로 파악해.
파악 끝나면 "내가 이해한 앱 동작"을 3-5줄로 요약해서 나한테 먼저 보여줘.
내가 OK 하면 다음 단계로 진행.

### 2단계: 테스트 계획 제안
아래 카테고리별로 뭘 테스트할지 목록으로 제안해:

- **순수 로직 (API mock 불필요)**
  - `validate_budget`: 700엔 미만 거부, 경계값, 음수, 문자열, None
  - `_parse_llm_json`: 정상 JSON, 앞뒤 잡음 있는 JSON, 깨진 JSON, 빈 문자열, 여러 JSON 블록
  - `is_trend_stale`: 파일 없음, 7일 이내, 7일 초과, 빈 파일
  - `load_prices` / `format_prices_for_prompt`: 정상 로드, 파일 없음, 깨진 JSON
  - `_get_cat_map`: 한/일 양쪽 키 일치 확인
  - `save_cart_markdown`: 디렉토리 자동 생성, 파일명 포맷, 내용 무결성
  - `apply_health`: add/remove 동작, 중복 처리, 빈 리스트
  - `display_preferences`: 언어별 필드 표시

- **API mock 기반 통합 테스트**
  - 쇼핑 에이전트: `price_est`, `price_source` 필수 필드 검증
  - 건강 에이전트: `comment` 2문장 구조, `add`/`remove` 필드 동작
  - 키워드 생성: 7개 반환, 타입별(트렌드/반찬/취향/랜덤) 분류
  - 언어 엄격 분리: 한국어 모드 응답에 일본어 섞이지 않는지
  - 에이전트 체인 end-to-end (전부 mock)

- **일본어 모드 전용 검증** (한국어 모드와 대칭)
  - 언어 선택 "2" 입력 시 모든 출력이 일본어
  - trend_jp.md / preferences_jp.md 로드 확인
  - 카테고리 헤더가 일본어 매핑 사용 (_get_cat_map("jp"))
  - 쇼핑/건강 에이전트 응답이 일본어인지 (mock 응답도 일본어로)
  - 합계 요약 문구 일본어 ("合計" 등)
  - carts/*.md 저장 시 일본어 인코딩 (UTF-8) 깨짐 없음
  - 일본어 모드 응답에 한국어 섞이지 않는지 검증

- **실제 API 스모크 테스트 (비용 발생, 의도적)**
  - 전체 플로우 한국어 모드 1회 (예산 1000엔, 키워드 1개 선택)
  - 전체 플로우 일본어 모드 1회 (예산 1000엔, 키워드 1개 선택)
  - 둘 다 `@pytest.mark.smoke`로 마킹, 기본 실행에서 제외
  - 각 테스트는 응답 언어가 모드와 일치하는지 검증 포함

- **Edge cases**
  - prices.json 없을 때 종료
  - carts/ 디렉토리 없을 때 자동 생성
  - 예산이 딱 700엔 (경계)
  - LLM이 가격 참고표 밖 상품 반환
  - 건강 에이전트가 오리지널에 없는 상품 remove 요청

계획 보여주고 내 승인 기다려.

### 3단계: 구현
승인 후 pytest로 구현. 다음 규칙 준수:

- `tests/` 폴더에 `test_*.py`로 분리 (단위/통합/스모크)
- Anthropic API는 `unittest.mock` 또는 `pytest-mock`으로 전부 모킹 (스모크 제외)
- mock 응답은 실제 에이전트가 반환할 법한 JSON 구조로 현실감 있게
- fixture로 prices.json, preferences, trend 파일 테스트용 임시 생성
- `tmp_path` fixture 적극 활용해서 파일 시스템 격리
- 한/일 양쪽 로케일 parametrize로 돌리기
- 스모크 테스트는 `pytest.mark.smoke` + `pyproject.toml` 또는 `pytest.ini`에 마커 등록, 기본 실행에서 제외
- 테스트 실행법을 `tests/README.md`에 적기:
    - `pytest` (기본, mock만, 무료)
    - `pytest -m smoke` (실제 API 호출, 비용 발생 경고)

### 4단계: 실행 및 리포트
구현 끝나면 `pytest -v` 돌리고 결과 보여줘.
실패한 게 있으면:
1. 테스트가 잘못된 건지, 앱 코드 버그인지 판단
2. 앱 코드 버그면 고치지 말고 나한테 먼저 보고
3. 테스트 문제면 수정

마지막에 `pytest --cov=shop` 커버리지 결과도 같이.

## 제약
- 내 승인 없이 shop.py 수정 금지 (버그 발견해도 보고만)
- 스모크 테스트 제외한 모든 테스트는 mock 사용
- Windows + PowerShell 환경 고려 (경로 구분자, 인코딩 등)
- `.env`의 ANTHROPIC_API_KEY는 스모크 테스트에서만 필요
- 모든 언어 의존 테스트는 ko/jp 양쪽 parametrize 필수
```

## 그 다음 단계별 대응 가이드

Claude Code가 프롬프트 받으면 이 순서로 진행할 것:

**→ 1단계 끝** — "내가 이해한 앱 동작" 요약 보여줌
  - 사용자가 할 일: 읽고 맞으면 "OK 2단계 진행해", 틀린 거 있으면 지적

**→ 2단계 끝** — 테스트 계획 목록 보여줌
  - 사용자가 할 일: 빠진 거/과한 거 없나 보고 "좋아 구현해" 또는 "X는 빼고 Y 추가해"

**→ 3단계 중** — 파일 만들면서 진행. 중간에 질문할 수도 있음
  - 사용자가 할 일: 질문 나오면 답해주기. shop.py 건드리려 하면 막기.

**→ 4단계 끝** — `pytest -v` 결과 + 커버리지 보여줌
  - 사용자가 할 일: 실패 있으면 같이 판단, 다 통과하면 스모크 돌릴 차례

## 4단계 이후: 스모크 테스트

mock 테스트 다 통과한 거 확인하면:

```
이제 스모크 테스트 돌려줘: pytest -m smoke -v
실제 API 호출 나가니까 결과 꼼꼼히 보여줘 — 
응답 내용, 언어 일치 여부, 토큰 사용량(가능하면).
```

여기서 한/일 각 1번씩 실제 호출 나감.
