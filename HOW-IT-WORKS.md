## 1. 한 줄 요약

일본 넷슈퍼에서 뭘 살지 고민하는 사람들을 위한 CLI 장바구니 생성기.<br/>
일본 거주 한국인 혹은 일본인을 주 사용자로 함.<br/>
Anthropic API 기반 에이전트 체인이 '트렌드 수집(7일 캐시) → 키워드 추천 → 장바구니 생성 → 건강 검토'를 해줌.

## 2. 앱 동작 흐름 — 이야기로 풀어본다면

**퇴근 후 냉장고 앞**

월요일 밤 11시. 냉장고 문을 열었더니 달걀 2개, 케첩, 그리고 정체불명의 소스 병 하나. 한숨이 절로 나온다.

"…장 봐야 하는데."

손가락이 무겁다. 뭘 살지 생각하는 것 자체가 일이다.

그때 터미널을 켠다. `python shop.py`.<br/><br/>


**1막. 일단 아무 생각 없이 시작**

```
언어 선택 / 言語選択
  [1] 한국어  [2] 日本語
→
```

`1`. 생각 없이.

```
이번 주 예산 (엔, 배송비 제외, 세전):
→
```

`5000`. 역시 생각 없이.<br/><br/>


**2막. 트렌드 에이전트 등장**

무대 뒤에서 트렌드 에이전트가 조용히 웹을 뒤진다. 눈치 빠른 스타일. <br/>
시키지 않아도 "혹시 트렌드 파일이 8일이나 됐네요, 제가 좀 다녀올게요" 하고 이미 나가있다. <br/>
5분 뒤 돌아와서 `trend_kr.md`를 갱신하고 자기 자리로 돌아간다.<br/><br/>


**3막. 키워드 에이전트의 제안**

키워드 에이전트가 오늘의 메뉴판을 내민다.

```
💡 오늘의 추천 키워드 (4월 14일 월요일, 봄)

  1. 지친 월요일에
  2. 봄나물 한 상
  3. 새 생활 응원
  4. [🔥 트렌드] 이번 주 트렌드 식품
  5. [🍱 반찬/냉동] 이번 주 추천 반찬·냉동식품
  6. [📌 내 취향] 내 취향 기반 추천
  7. [🎲 알아서] 알아서 해줘
```

"봄나물 한 상"은 무슨. 지금 기력이 없어서 봄나물 씻을 힘도 없다. `1`. 지친 월요일에. 이게 딱 맞다.<br/><br/>


**4막. 쇼핑 에이전트, 콧노래 부르며 등장**

쇼핑 에이전트는 자신감이 넘친다. 대표 식품 가격 참고표를 옆에 끼고, 당신의 취향까지 꿰차고 있다. <br/>
목표는 명확하다. 4500엔~5500엔 사이. 10% 오차 이내.

"지친 월요일이요? 알죠. 데우기만 하면 되는 거 위주로 채워드릴게요."

냉동 볶음밥, 레토르트 카레, 냉동 교자, 컵라면 두 개, 계란, 바나나, 요구르트… 빠르게 목록이 채워진다. <br/>
합계 4,820엔. 만족스럽다는 듯 JSON을 제출하고 물러선다.<br/><br/>


**5막. 건강 에이전트, 혀를 차며 등장**

건강 에이전트가 목록을 훑는다. 딱 1초.

"…탄수화물 천국이네요."

쇼핑 에이전트가 발끈한다. "예산 안에 맞췄잖아요!"

"냉동 볶음밥에 레토르트 카레에 컵라면이 두 개예요. 채소가 바나나밖에 없고요."

"바나나도 과일이잖아요."

"…시금치 한 봉지 추가하고, 컵라면 하나 빼겠습니다."

쇼핑 에이전트가 투덜댄다. "그럼 예산이…"

"140엔 줄어요. 괜찮아요."

둘이 실랑이하는 동안 프로그램은 건강 에이전트 말대로 목록을 수정한다. <br/>
삭제된 건 ❌, 줄어든 건 ➖, 추가된 건 ➕. 깔끔하다.<br/><br/>


**6막. 결과물**

```
💰 합계 4,820엔 / 예산 5,000엔 (오차 -180엔)
📄 저장됨: carts/cart_2026-04-14_230847.md
```

총 API 호출 4번. 트렌드 갱신이 없었다면 3번. 에이전트들은 서로 얼굴 한 번 본 적 없다. 코드가 중간에서 메모를 전달했을 뿐이다.

당신은 목록을 갖고 넷슈퍼 웹사이트에 접속한다. 생각할 게 없다. 그냥 따라 담으면 된다.

냉장고 문을 열 때의 그 한숨이, 조금 가벼워졌다.

## 3. 아키텍처

```
사용자 입력 (언어, 예산, 키워드)
       │
       ▼
┌─────────────────┐   web_search
│  트렌드 에이전트  │──────────────→ trend_{kr,jp}.md
└─────────────────┘
       │
       ▼
┌─────────────────┐   claude-sonnet-4-6
│  키워드 에이전트  │──→ AI 3개 + 하드코딩 4개 = 최대 7개 키워드
└─────────────────┘
       │
       ▼  (사용자가 키워드 선택)
┌─────────────────┐   prices.json + preferences_{kr,jp}.md
│  쇼핑 에이전트    │──→ {"comment": "감상", "items": [...]}
└─────────────────┘
       │
       ▼
┌─────────────────┐   검증/재시도 로직 포함
│  건강 에이전트    │──→ {"comment": "조언", "add": [...], "remove": [...], "reduce": [...]}
└─────────────────┘
       │
       ▼
  apply_health()로 병합 → 화면 출력 + carts/*.md 저장
```

**핵심 포인트**: 각 에이전트는 독립된 API 호출이다. 서로의 대화를 기억하지 못한다.
에이전트 간 정보 전달은 코드 레벨에서 함수 인자로 한다 (쇼핑 결과 → health_check의 items 파라미터).

### 연결 관계

- `shop.py`가 `prices.json`을 읽어서 시스템 프롬프트에 끼워넣는다
- `preferences_*.md`도 마찬가지. LLM이 취향을 반영할 수 있게
- `trend_*.md`는 키워드 생성 에이전트에 주입. 7일 지나면 `fetch_and_save_trend()`가 웹검색으로 자동 갱신
- 결과는 `carts/cart_YYYY-MM-DD_HHMMSS.md`로 저장

### pyproject.toml

**반드시 프로젝트 루트에 있어야 한다.** pytest가 실행될 때 루트에서 이 파일을 자동으로 찾아 읽는다.

이 파일이 하는 일을 비유하면, **pytest한테 미리 귓속말로 규칙을 알려주는 것**이다.

- `markers = ["smoke: real API calls (costs money)"]` — 테스트 코드에는 `@pytest.mark.smoke`라는 태그가 붙은 테스트들이 있다. 이 태그를 미리 등록해두지 않으면 pytest가 "이게 무슨 태그야?" 하고 경고를 낸다. 그래서 "smoke는 내가 만든 태그야, 실제 API를 호출하는 비싼 테스트에 붙인 거야"라고 등록해둔다.
- `addopts = "-m 'not smoke'"` — `pytest`를 그냥 실행하면 smoke 태그가 붙은 테스트는 자동으로 제외된다. 실수로 유료 API를 호출하는 일을 막기 위해서다. 실제 API를 테스트하고 싶을 때만 `pytest -m smoke`라고 명시적으로 실행한다.

tests 폴더 안으로 옮기면 pytest가 찾지 못해 두 설정 모두 무시된다.

### 앱 실행 중 자동으로 생기는 파일들

직접 만든 게 아닌데 폴더에 생겼다면 정상이다. 지워도 다음 실행 시 다시 생긴다.

| 파일/폴더 | 용도 | 언제 생기나 | 지워도 되나 |
|---|---|---|---|
| `carts/` | 생성된 장바구니 목록 저장 | `shop.py` 실행할 때마다 | O (과거 목록이 필요 없다면) |
| `trend_kr.md`, `trend_jp.md` | 일본 슈퍼 트렌드 캐시 | 처음 실행 시, 또는 7일이 지났을 때 | O (다음 실행 시 재생성됨) |
| `__pycache__/`, `*.pyc` | Python 바이트코드 캐시 | Python이 처음 실행될 때 | O (다음 실행 시 재생성됨) |
| `.pytest_cache/` | pytest 실행 결과 캐시 | `pytest` 실행 시 | O |
| `.coverage` | 테스트 커버리지 측정 결과 | `pytest --cov` 실행 시 | O |

### .env — 직접 만들어야 하는 파일

앱 실행 전에 프로젝트 루트에 `.env` 파일을 직접 만들고 Anthropic API 키를 넣어야 한다.

```
ANTHROPIC_API_KEY=sk-ant-...
```

API 키는 [Anthropic 콘솔](https://console.anthropic.com)에서 발급받는다. **이 파일은 절대 외부에 공유하거나 git에 올리지 않는다.**

## 5. 기술 결정과 이유

### "왜 단일 파일(shop.py)인가?"
CLI 도구 특성상 복잡한 모듈 분리보다 한 파일에서 흐름을 따라가기 쉬운 구조가 유지보수에 유리하다.
함수 단위로 충분히 분리되어 있어서 테스트도 가능하다.

### "왜 시스템 프롬프트가 전부 일본어인가?"
사용자가 이용하는 넷슈퍼는 일본을 기반으로 한다. 상품명, 카테고리, 가격 기준이 모두 일본어다.
LLM에게 일본어로 사고하게 한 뒤 출력만 한국어/일본어로 분리하는 게 더 정확하다.
단, 이 때문에 한국어 모드에서 일본어가 섞이는 부작용이 있어서 명시적 분리 지시가 필요했다.

### "왜 가격을 전부 세전으로?"
넷슈퍼 표시가 세전이고, 일본 식품 소비세(8%)는 일률적이라 LLM이 혼동할 여지를 줄이기 위해.
배송비도 별도. 단순할수록 LLM이 잘 따른다.

### "왜 prices.json이 필요한가?"
LLM은 가격을 잘 모른다. "바나나 1000엔"같은 결과가 나올 수 있다.
참고표를 주면 `price_source: "reference"`로 정확한 가격을 쓰고, 없는 건 `"inferred"`로 추정한다.
82개 품목으로 확장한 이유: LLM이 참고표 안의 "안전한" 상품만 반복 추천하는 경향이 있어서, 풀을 넓혀 다양성을 확보했다.

### "왜 reduce 필드를 추가했는가?"
건강 에이전트가 "하겐다즈 4개→2개"로 줄이고 싶을 때, remove만 있으면 4개를 통째로 날린다.
reduce 필드로 수량 감소를 별도 처리하면 예산 손실이 절반으로 줄고, UI에서도 ➖로 구분 가능.

## 6. 버그 해결 과정과 교훈

### 한국어 모드에 일본어 섞임 ("카ット 샐러드")
- **원인**: 시스템 프롬프트가 전부 일본어 + 입력도 jp 필드만 전달
- **해결**: (1) 프롬프트에 "韓国語モードではカタカナ禁止" 명시 + 한국어 예시 추가, (2) health_check 입력에 kr 번역 병기
- **교훈**: LLM은 **입력 언어**에 끌려간다. 출력 언어를 바꾸려면 입력에도 해당 언어를 섞어줘야 한다.

### 예산 대비 합계가 너무 낮음 (64~77%)
- **원인**: "예산 초과 금지" 지시만 있고 "예산에 가깝게 채워라"는 지시가 없었음
- **해결**: (1) 90~110% 목표 명시, (2) 유저 메시지에 구체적 금액 범위 제공 (`目標合計: 7200〜8800円`), (3) 건강 에이전트에 예산 보전 지시
- **교훈**: LLM은 암산을 못한다. 퍼센트 대신 **구체적 숫자**를 줘야 한다.

### 건강 에이전트의 comment/action 불일치
- **원인**: LLM이 comment 텍스트와 JSON action을 순서대로 별개 생성. comment 작성 시점엔 "빼야겠다"고 했다가, action 채울 때 잊거나 형식이 헷갈려 빈 배열을 넣는 현상.
  ```json
  // 불일치 예시: 말은 하는데 실제 동작이 없음
  { "comment": "컵라면을 빼고 채소를 추가하세요.",
    "remove": [],
    "add": [] }
  ```
- **해결**: (1) `_comment_suggests_changes()`로 comment에 "빼", "줄이" 등 액션 단어 감지, (2) action이 비어있으면 원본 상품명 목록을 붙여 1회 재시도 (`_validate_health_advice`)
- **교훈**: 프롬프트에 "일치시켜라"고 써도 LLM이 지킨다는 보장이 없다. **코드에서 직접 검증하고 재시도**하는 로직이 필수.

### max_tokens 부족으로 JSON 잘림
- **원인**: prices.json 82개 확장 → 시스템 프롬프트 길어짐 → 응답 공간 부족
- **해결**: 쇼핑 에이전트 2048→4096, 건강 에이전트 512→1024
- **교훈**: 입력이 길어지면 출력 한도도 같이 늘려야 한다. 기능 추가할 때마다 체크.

### apply_health가 동일 이름 전부 삭제
- **원인**: `item["jp"] in removed`는 이름 존재 여부만 체크. 카트에 동일 이름 아이템이 여러 개면 전부 삭제.
  ```python
  # 버그 예시: 컵라면 3개 중 1개만 빼라고 했는데
  removed = set(["カップ麺 1個"])   # 이름만 기억
  for item in items:
      if item["jp"] in removed:    # 3개 전부 매칭 → 전부 삭제
  ```
- **해결**: count 기반 삭제 — `remove` 배열의 출현 횟수만큼만 카운터를 감소시켜 처리
  ```python
  # 수정 후: 카운터가 0이 되면 나머지는 건드리지 않음
  remove_counts = {"カップ麺 1個": 1}
  for item in items:
      if remove_counts.get(jp, 0) > 0:
          remove_counts[jp] -= 1  # 1개만 삭제, 나머지 2개는 유지
  ```
- **교훈**: `x in set` 방식의 포함 여부 확인은 중복 아이템에서 위험하다. **몇 번 삭제할지 횟수를 세야** 한다.

## 7. 주의사항과 함정

### Windows 환경
- **콘솔 인코딩**: Windows 터미널은 한국어 세팅이면 `cp949`(한글 전용), 일본어 세팅이면 `cp932`(일본어 전용)가 기본이다. 두 방식 모두 이모지(🛒)를 지원하지 않아서, 그냥 `print()`하면 "이 글자 어떻게 출력하는지 모르겠어요" 에러(`UnicodeEncodeError`)가 난다. 그래서 `shop.py` 첫 줄에서 "앞으로 UTF-8로 출력해"라고 강제로 바꿔준다(`sys.stdout.reconfigure(encoding="utf-8")`).
- **경로 표기**: 파일 경로 구분자가 Windows는 `\`, Mac/Linux는 `/`라 문자열로 직접 쓰면 OS마다 달라진다. `Path` 객체를 쓰면 OS에 맞게 알아서 바꿔주므로 신경 쓸 필요가 없다.

### API 호출 특성
- **매 호출이 독립적**: 이전 대화를 기억하지 못한다. "지난번과 다른 상품"같은 지시는 의미 없다.
- **비용 발생**: 키워드(1회) + 쇼핑(1회) + 건강(1~2회, 재시도 포함) = 최소 3회, 최대 4회 API 호출.
- **Rate limit**: 웹검색 포함 호출은 토큰을 많이 먹어서 분당 한도에 걸리기 쉽다.

### LLM의 습성
- **참고표 선호**: 가격이 확실한 reference 상품을 좋아하고, inferred는 피하려 한다. 강제 지시("2~3할은 참고표 밖") 필요.
- **고정 패턴**: 예시에 적힌 상품(だし巻き卵, とんかつ)을 매번 먼저 넣는다. 고정 예시 제거로 해결.
- **암산 불가**: 20개 상품 가격을 합산해서 예산에 맞추는 건 기대하면 안 된다. 코드에서 목표 금액을 계산해서 넘겨줘야 한다.

## 8. 테스트 전략

```
pytest          → mock 테스트 110개 (무료, 빠름)
pytest -m smoke → 실제 API 2회 (한/일 각 1회, 몇 센트)
pytest --cov=shop → 커버리지 리포트
```

### 설계 원칙
- **API mock으로 기본**: `unittest.mock.patch.object(shop.client.messages, "create")`. 실제 호출 없이 JSON 파싱, 함수 로직 검증.
- **스모크는 최소한**: 한국어 1회 + 일본어 1회. 응답 구조와 언어 일치만 확인.
- **모듈 임포트 문제**: shop.py가 로드 시 `ANTHROPIC_API_KEY`를 체크한다. conftest.py에서 `load_dotenv()` → 없으면 더미 키 세팅.
- **파일 시스템 격리**: `tmp_path` + `monkeypatch.chdir` 사용. 실제 carts/ 디렉토리 오염 방지.

## 9. 좋은 엔지니어링 실천

### "프롬프트도 코드다"
시스템 프롬프트 변경은 코드 변경과 같다. 수정 후 반드시 실행해서 결과를 확인한다.
단위 테스트는 프롬프트 효과를 검증할 수 없다 — 실제 돌려봐야 안다.

### "LLM을 믿되, 검증하라"
LLM의 JSON 출력은 항상 깨질 수 있다 (`_parse_llm_json`의 코드펜스 제거, raw_decode 등).
LLM의 comment와 action이 일치하리라 기대하지 말고 코드에서 검증하라.

### "가격 데이터는 수작업이 가장 정확하다"
웹검색 스크립트(`update_prices.py`)로 자동화를 시도했지만 성공률이 낮았다 (33개 중 3개만 성공).
결국 Gemini에게 상식 가격을 물어보고 수동 머지하는 게 더 빠르고 정확했다.
자동화가 항상 답은 아니다.

### "한/일 대칭은 강제 규칙"
한쪽 언어만 구현하면 반드시 나중에 빠뜨린다.
CLAUDE.md에 "한쪽만 구현 금지" 규칙을 명시하고, 테스트도 `@pytest.mark.parametrize("lang", ["ko", "ja"])`로 양쪽을 강제.
